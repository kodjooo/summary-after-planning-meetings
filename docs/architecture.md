# Архитектура AI Meeting Assistant Bot

## 1. Назначение системы

Сервис представляет собой Telegram-бота, который принимает аудиозаписи встреч, передаёт их в фоновую обработку, делает транскрипцию через OpenAI Whisper API и анализирует результат через OpenAI Responses API. Пользователь получает структурированный протокол встречи прямо в Telegram.

## 2. Основные компоненты

### Telegram Bot
- Реализован на `aiogram`.
- Работает через long polling в контейнере `bot`.
- Обрабатывает команды `/start` и `/help`.
- Принимает сообщения типов `voice`, `audio`, `document`.
- Валидирует расширение файла и отсеивает только действительно крупные входы выше общего лимита приложения.
- Если Telegram Bot API не дает скачать большой файл, бот генерирует одноразовую ссылку на резервную web-загрузку.

### Redis
- Используется как брокер и backend для `Celery`.
- Доступен только внутри сети `docker compose` и не публикуется наружу на хост-машину.

### Celery Worker
- Работает в контейнере `worker`.
- Выполняет длительные задачи без блокировки Telegram-бота.
- Подписан на очередь `meetings`, в которую бот отправляет задачи обработки встреч.
- Обрабатывает `process_meeting_task` для полного pipeline обработки встречи.

### Web fallback
- Работает в контейнере `web` на `FastAPI`.
- Отдает одноразовую HTML-форму загрузки большого файла по токену из Redis.
- После загрузки ставит локальный файл в тот же Celery pipeline, что и Telegram-вложения.
- Использует только временное хранение: загруженный файл переносится в рабочую директорию job и удаляется после обработки.

### Модуль аудиообработки
- Скачивает файлы из Telegram во временную директорию контейнера.
- Конвертирует `ogg`, `m4a` и `aac` в `wav` через `ffmpeg`.
- Объединяет несколько `voice`-сообщений в единый `wav`.
- Если итоговый файл слишком большой для Whisper, автоматически режет его на несколько `.wav`-частей через `ffmpeg`.
- Очищает временные файлы после завершения обработки.

### OpenAI-интеграции
- `audio.transcriptions` с моделью `whisper-1` используется для транскрипции.
- `Responses API` используется для анализа содержимого встречи.
- Для длинных транскрипций используется схема chunking -> промежуточные summary -> финальный анализ, порог чанкинга настраивается через `TRANSCRIPT_CHUNK_SIZE`.
- Системный и пользовательский промпты вынесены в отдельные файлы, чтобы их можно было менять без правки бизнес-логики.

### Модуль форматирования результата
- Преобразует структурированный ответ в Telegram Markdown.
- Всегда отправляет краткое сообщение в чат и прикладывает полный результат отдельным `.txt`-файлом.

## 3. Поток данных

1. Пользователь отправляет `voice`, `audio` или `document` в Telegram.
2. Бот валидирует расширение и размер файла.
3. Для `voice`, `audio` и `document` бот сразу ставит задачу в Celery без дополнительного ожидания.
4. Worker скачивает файл из Telegram.
5. Если Telegram не дает скачать слишком большой файл, бот отправляет пользователю одноразовую ссылку на web-загрузку.
6. Пользователь загружает файл через web-форму, и web-сервис ставит его в тот же Celery pipeline.
7. Worker конвертирует и при необходимости объединяет аудио.
8. Если итоговый аудиофайл превышает безопасный лимит Whisper, worker режет его на части и транскрибирует каждую часть отдельно.
9. Worker склеивает транскрипции частей в единый текст.
10. Worker отправляет транскрипцию в Responses API и получает структурированный анализ.
11. Worker форматирует результат и отправляет его пользователю в Telegram.
12. Временные файлы удаляются, итоговый статус логируется.

## 4. Статусы обработки

Внутренние статусы задачи:
- `queued`
- `downloading`
- `preprocessing`
- `transcribing`
- `analyzing`
- `sending`
- `done`
- `failed`

Пользовательские статусы:
- `Файл получен. Начинаю обработку.`
- `Начинаю транскрипцию.`
- `Анализирую встречу.`
- `Готово.`

## 5. Конфигурация

Основные переменные окружения:
- `TELEGRAM_BOT_TOKEN`
- `OPENAI_API_KEY`
- `OPENAI_ANALYSIS_MODEL`
- `OPENAI_REASONING_EFFORT`
- `OPENAI_TRANSCRIPTION_MAX_FILE_SIZE_MB`
- `MAX_FILE_SIZE_MB`
- `WEB_BASE_URL`
- `UPLOAD_TOKEN_TTL_SECONDS`
- `REDIS_URL`
- `TEMP_DIR`
- `TRANSCRIPT_CHUNK_SIZE`
- `BOT_STATUS_POLLING_TIMEOUT`

Все настройки читаются через `pydantic-settings`. `.env.example` содержит шаблон, `.env` используется только локально и не должен попадать в GitHub.

## 6. Ограничения MVP

- Постоянная база данных не используется.
- Постоянное файловое хранилище не используется, только временная директория внутри контейнера.
- Поддерживаются только форматы `ogg`, `mp3`, `m4a`, `wav`, `aac`.
- Запуск проекта поддерживается только через Docker Desktop и `docker compose`.

## 7. Надёжность и обработка ошибок

- Неподдерживаемые файлы отклоняются до постановки в очередь, а большие допустимые аудио режутся на части перед Whisper.
- Если Telegram API не позволяет скачать файл из чата, используется резервная web-загрузка по одноразовой ссылке.
- Пустые и повреждённые файлы отсекаются на этапе аудиообработки.
- Для временных ошибок брокера и сетевых вызовов используется retry-политика Celery.
- При ошибке пользователь получает единое сообщение: `Не удалось обработать файл. Попробуйте снова.`

## 8. Тестовое покрытие

В проекте предусмотрены тесты для:
- валидации входных файлов;
- fallback-токенов и web-формы загрузки;
- парсинга и форматирования результата анализа;
- обработки длинной транскрипции;
- smoke-проверки структуры `docker-compose.yml`.
